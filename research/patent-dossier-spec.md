# Patent Dossier — Format & Data Spec

> Section-by-section spec for the Patent Dossier full-tab report (`patent.html?number=...`)
> Companion to `patent-dossier-architecture.md` (which covers *why* and *where*; this covers *what fields*).
>
> Date: 2026-05-10
> Mockup: `patent.html` (populated with US10867416B2 — Adobe "Harmonizing composite images using deep learning")

---

## Scope

**Phase 1 (this spec):** every section that can be populated from a Google Patents page scrape. Renders today with no new data integration.

**Phase 2 (placeholders only):** sections that require USPTO ODP (file wrapper, office actions, examiner stats). Listed here so the layout is future-proof, but they ship empty/hidden in v1.

**Data sourcing legend used below:**
- ✅ **GP** — extractable from the Google Patents page (current scrape pipeline)
- ⚠️ **GP partial** — some signal in GP, supplement later
- ⏳ **ODP** — USPTO Open Data Portal API, Phase 2
- 🧠 **AI** — generated by Gemini from the patent text

---

## Surface 1 — Side panel (Patent tab)

The side-panel surface is a **summary + launch** UI, not a substantial report. ~400px wide.

| Element | Source | Notes |
|---|---|---|
| Patent number header | URL or auto-detected from active GP tab | Editable input |
| Status chip | ✅ GP | `Active`, `Lapsed`, `Expired`, `Pending`, color-coded |
| Family count chip | ✅ GP | `Family: 7` |
| Claims count chip | ✅ GP | `Claims: 24` |
| Cite counts chip | ✅ GP | `Cites: 142↓ 38↑` (forward / backward) |
| **Open full dossier** button | n/a | `chrome.tabs.create({ url: 'patent.html?number=...' })` |
| **Recent dossiers** list | local | Last ~5 patents looked up this session |
| Optional inline expanders | ✅ GP | "Quick family", "Quick claims" — show top items without leaving the side panel |

Hybrid pattern: chips answer 80% of "what am I looking at" in 2 seconds; the full tab is the deliverable.

## Phase 2 sections (hidden in v1)

These appear in the architecture doc but are not in scope until USPTO ODP integration:

- File wrapper viewer (full prosecution history)
- Office Action analyzer
- Examiner statistics
- IDS generation
- Per-family-member legal events

**Recommendation:** render these as hidden `<details>` blocks in the static mockup, so reviewers can preview the eventual fully-loaded report. They flip on once the ODP pipeline lands.

## Layout rules

- **Max width** 900px (matches sample-report)
- **Numbered sections** with circular section-number badges (`.section-number`)
- **Section intro** in `.section-intro` (blue left border, light grey bg) explaining what's in the section
- **Color tokens:** reuse the `:root` palette from `sample-report.html` (`--primary`, `--accent`, `--green`, `--yellow`, `--red`, `--border`, `--bg-light`)
- **Print media query:** ship with `@media print { ... }` so the same HTML exports as a PDF cleanly
- **No JS required for the v1 mockup** — every interactive feature is `<details>` or a static placeholder so the page renders standalone

## Open questions to decide before implementation

1. **Real-time vs cached fetch.** Each dossier requires a GP page scrape. Cache window? (Recommend: 24h for legal status, 7d for everything else.)
2. ~~**AI summary trigger.** Auto-generate on page load (costs credits) or on user click ("Generate summary →")? Recommend the latter for free tier.~~ **Decided 2026-05-11:** auto-generate on full-tab dossier load, bundled with the 3-credit dossier purchase (no additional charge). Rationale: every credit-paying user is already committed to a full-dossier view; the 30-day summary cache means re-opens are free and instant.
3. **Family graph rendering.** Light layouts (≤5 members) use a flat table; bigger families want a tree. Threshold?
4. **Persistence.** When do shareable URLs ship — same release as v1, or a Phase 2 follow-up? Architecture doc favors Phase 2.
5. **Side-panel chips.** Do chips fetch lazily from the GP tab the user has open, or always trigger a fresh server-side fetch? (Lazy is faster but breaks when the user isn't on a GP page.)
